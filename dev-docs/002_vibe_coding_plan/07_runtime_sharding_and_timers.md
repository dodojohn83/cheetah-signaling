# 07 设备运行时、分片与定时器

## 1. 目标

实现单节点可轻量运行、集群可水平扩展的设备运行时。运行时必须避免“每设备一个常驻 Tokio task”的百万级资源放大，采用固定分片执行器、惰性会话和分层定时轮。

## 2. 组件设计

在 `crates/runtime` 中实现：

- `ShardRuntime`：固定数量分片及其邮箱。
- `DeviceActor`：仅在设备活跃期间存在的轻量状态对象，不等同于独立 task。
- `ShardRouter`：按稳定哈希把 `DeviceKey` 映射到分片。
- `TimerWheel`：心跳、注册过期、命令超时和重试定时器。
- `SessionRegistry`：保存本节点协议连接/事务句柄。
- `AdmissionController`：限制邮箱、连接和并发命令。

## 3. 容量与默认值

默认分片数为 `max(16, cpu_count * 4)`，只能在首次部署时配置；运行中变更需要显式迁移。每分片使用有界 MPSC 邮箱，默认容量 8192。配置必须允许 ARM 设备下调，但禁止无界通道。

设备非活跃时只保留数据库状态；注册、心跳、事务或命令到来时加载到分片，达到空闲阈值后回收。

## 4. 实现任务

### RUN-001：分片路由

- [ ] 哈希输入包含 `tenant_id + device_id`，使用固定算法和版本号。
- [ ] 提供黄金向量测试，确保升级编译器或依赖后映射不漂移。
- [ ] 同一设备的消息在单分片内顺序执行。
- [ ] 跨设备操作不得持有两个 actor 的互斥锁。

### RUN-002：消息执行循环

- [ ] 定义 `RuntimeMessage`：协议事件、领域命令、计时器、所有权变化、关闭信号。
- [ ] 单次轮询限制每设备连续处理条数，避免热点设备饿死其他设备。
- [ ] 捕获 handler panic，在指标中记录并隔离故障分片；进程级 panic 策略按环境配置。
- [ ] 邮箱满返回明确过载错误，并由入口层采用退避或拒绝策略。

### RUN-003：定时器

- [ ] 采用至少两级时间轮，短周期精度不低于 100ms，长周期不为每项创建 Tokio sleep。
- [ ] 定时项包含 `timer_id`、`device_key`、`kind`、`deadline`、`generation`。
- [ ] 取消操作通过 generation 防止旧定时事件误触发。
- [ ] 重启后的关键 deadline 从持久化数据重建。

### RUN-004：生命周期与优雅停机

- [ ] 启动顺序：配置 → 存储 → 消息系统 → 所有权 → runtime → 协议监听 → 北向 API。
- [ ] 停机先关闭接入和 readiness，再停止接收新命令。
- [ ] 在宽限期内完成在途事务、刷新 outbox、释放所有权并关闭媒体会话。
- [ ] 超时后强制终止并记录未完成项数量。

### RUN-005：背压和资源预算

- [ ] 分别限制全局、租户、协议、设备四级并发。
- [ ] 为注册、心跳、控制命令设置不同优先级；心跳风暴不得阻断停机信号。
- [ ] 暴露邮箱深度、处理延迟、actor 数、timer 数和拒绝数指标。

## 5. 测试与基准

- [ ] 使用暂停时间的 Tokio 测试验证百万定时项调度，不依赖真实等待。
- [ ] 100 万离线设备不得创建 100 万 task。
- [ ] 热点设备持续压测时，普通设备消息 P99 不超过目标阈值两倍。
- [ ] 模拟分片 handler 失败，其他分片继续服务。
- [ ] ARM64 构建运行内存基线测试，空载 RSS 与每活跃设备增量写入基准报告。

## 6. 验收标准

- 所有通道有界，所有拒绝可观测。
- 设备内事件保持顺序；跨节点顺序由 owner epoch 和幂等约束保证。
- runtime 不包含 GB28181/ONVIF 特有类型。
