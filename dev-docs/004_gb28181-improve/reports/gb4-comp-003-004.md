# GB4-COMP-003/004：GB28181 SDP/Broadcast/MediaStatus 受控 override 与 provenance fixture

## 任务目标

在 GB4-COMP-001/002 的 `CompatibilityProfile` / `ProfileResolver`（exact match、revision pinning、capability 枚举）基础上：

- **GB4-COMP-003**：扩展 profile schema 与 resolver，覆盖 SDP/Broadcast/MediaStatus 三类媒体协商 override —— 允许的 SDP payload/attribute 模式、broadcast 地址来源、MediaStatus 响应归一化；并保持 MediaPort 网络边界（信令面不解析或存储 RTP/RTCP/PS/TS/ES 负载，只通过类型化的地址/payload 协商）。
- **GB4-COMP-004**：为每个 workaround 增加 provenance fixture，记录样本输入、风险、regression 测试引用与移除条件。

所有 override 仅在匹配的 device/firmware profile 激活且能力显式启用时生效；默认保持严格行为。

## 实现位置

### domain 层（`crates/domain/cheetah-domain`）

- `src/compatibility.rs`（新增）：
  - `CompatibilityOverrides { sdp, broadcast, media_status }` 类型化 override 容器，带 `validate()` 上限校验（每个列表 ≤ 64 条，单条 ≤ 64 字节，非空）。
  - `SdpMediaOverride { allowed_payload_types, allowed_attribute_names }`：`allows_payload` / `allows_attribute` 只对显式列出的十进制 payload type 与 attribute name 返回 true。
  - `BroadcastAddressSource { MediaNode（默认）, SignalingHost }`（`#[non_exhaustive]`，`snake_case` serde）与 `BroadcastOverride { address_source }`。
  - `MediaStatusOverride { stopped_status_codes }`、`MediaStatusOutcome { Stopped, Unknown }`（`#[non_exhaustive]`）、常量 `MEDIA_STATUS_STOPPED_NOTIFY_TYPE = "121"`；`is_stopped` 识别 canonical `121` 与显式配置的 vendor code。
- `src/protocol_session.rs`：
  - `CompatibilityCapability` 新增 `SdpMediaOverride`（parser 映射 `"sdp_media_override"`）。
  - `CompatibilityProfile` 新增 `overrides: CompatibilityOverrides` 字段，`validate()` 调用 `overrides.validate()`。
  - 新增 capability 门控方法：
    - `sdp_payload_allowed`：baseline `0`/`8`/`96..=127` 接受；其他需 `SdpMediaOverride` 能力 + 显式列出。
    - `sdp_attribute_allowed`：baseline 属性名（`downloadspeed`/`filesize`/`username`/`password`）接受；未知名需 `SdpMediaOverride` + 显式列出。
    - `broadcast_address_source`：无 `Broadcast` 能力时返回 `MediaNode`；启用时返回配置来源。
    - `media_status_outcome`：无 `MediaStatus` 能力返回 `Unknown`；否则归一化 `121` 与 vendor stopped code 为 `Stopped`。
- `src/lib.rs`：`pub mod compatibility` 及类型 re-export。

### core 层（`crates/protocols/cheetah-gb28181-core`）

- `src/lib.rs`：re-export `BroadcastAddressSource`、`BroadcastOverride`、`CompatibilityOverrides`、`MediaStatusOutcome`、`MediaStatusOverride`、`SdpMediaOverride`，供 module 使用而不直接依赖 domain。

### module 层（`crates/protocols/cheetah-gb28181-module`）

- `src/media/mod.rs`：`MediaConfig` 增加 `compatibility: CompatibilityProfile`，把匹配到的活跃 profile 带入媒体协商，避免 override 变成全局行为。
- `src/media/handlers.rs`：对端 SDP 应答解析后调用 `validate_remote_sdp`，逐个 media format 走 `sdp_payload_allowed`、逐个 `SdpAttribute::Unknown` 走 `sdp_attribute_allowed`；命中不允许项时 `reject_and_teardown`（ACK + BYE + 移除会话 + 失败事件）。仅处理类型化元数据（SSRC、传输协议、端口、连接地址），不触碰媒体负载字节。
- `src/media/commands.rs`：`StartTalk`（语音对讲/广播）在 `broadcast_address_source() == SignalingHost` 时，将 SDP `c=` 连接地址锚定到信令主机地址，默认保持媒体节点地址。仅改变协商地址，不绑定媒体端口。
- `src/xml/media_status.rs`（新增）：`MediaStatusInfo` 与 `parse_media_status` / `extract_media_status`，只读取类型化 `NotifyType` 标识。
- `src/access.rs`：MESSAGE 分发新增 `MediaStatus` 分支，**仅在 `MediaStatus` 能力启用时**处理（否则回到 `UnsupportedCmdType` → 400），归一化后发出 `Gb28181Event::MediaStatusReceived`。
- `src/events.rs`：新增 `MediaStatusReceived { domain_id, device_id, source, sn, notify_type, outcome }` 事件。

## MediaPort 网络边界

- SDP 仅作为信令元数据处理：地址、端口、传输协议、SSRC、payload 标识与类型化属性可协商，RTP/RTCP/PS/TS/ES 字节从不解析或存储。
- Broadcast override 只改变 SDP 协商地址来源，不绑定媒体端口、不中转负载。
- MediaStatus 只读取类型化 `NotifyType`，不接触媒体负载。
- 架构审计 `python3 scripts/audit_architecture.py` 未新增任何 gb28181 相关的层次/依赖违规。

## Provenance fixture（GB4-COMP-004）

`testdata/gb28181/profiles/` 下每个 override 一对 `meta.toml` + 样本输入，均通过 `scripts/verify_gb4_fixtures.py`。`scripts/verify_gb4_fixtures.py` 扩展了 provenance 字段（`risk`、`regression_test`、`removal_criteria`）并**强制** profile 名以 `gb4-comp-003` / `gb4-comp-004` 开头的 fixture 提供这些非空字段；样本输入即 fixture 数据文件本身。

| Override | 样本输入 | meta 文件 |
|----------|----------|-----------|
| SDP payload/attribute | `sdp-media-override.txt` | `sdp-media-override.meta.toml` |
| Broadcast 地址来源 | `broadcast-address-source.txt` | `broadcast-address-source.meta.toml` |
| MediaStatus 归一化 | `media-status-normalization.xml` | `media-status-normalization.meta.toml` |

每个 meta 文件均记录：`risk`（风险）、`regression_test`（regression 测试引用）、`removal_criteria`（移除条件），并保持 synthetic 脱敏样本。

## 风险与移除条件

| Override | 风险 | 移除条件 |
|----------|------|----------|
| SDP payload/attribute | 放宽 allow-list 可能让媒体面无法处理的描述通过或掩盖畸形应答；范围受限：仅显式列出的 payload/attribute，其余保持严格，且不解析负载字节 | 目标固件退役，或设备确认仅发送 baseline payload（0/8/96-127）与 baseline 属性后移除 |
| Broadcast 地址来源 | 改变对讲/广播 SDP 连接地址仅影响协商地址，不会导致信令绑定媒体端口或中转负载；配置错误可能导致音频不通，故能力门控并默认媒体节点 | 目标对讲固件接受媒体节点地址，或设备型号退役后移除 |
| MediaStatus 归一化 | 将非标准 `NotifyType` 视为 stopped 可能过早结束会话；能力门控 + 显式 per-profile stopped-code 列表 + 仅读取类型化 `NotifyType` | 目标固件发送 canonical `121`，或设备型号退役后移除 |

## 附带修复：通道命令目标解析

Devin Review 指出基线分支的命令路由缺陷（本 PR 因 target 为 `main` 而携带完整 base diff）：`replace_catalog`（`apps/cheetah-signaling/src/gb_event_sink.rs`）在写入通道 metadata 时从未存储通道 GB28181 external id，而 `OwnerCommandHandler::resolve_gb_command`（`apps/cheetah-signaling/src/workers.rs`）通过 `metadata().get("external_id")` 读取该 id；由于内部 `ChannelId` 是 UUIDv5 单向哈希无法还原，读取恒为 `None`，导致通道级命令（PTZ/Preset 等）在多通道 NVR 上错误地发往设备自身 id。

修复：抽出 `build_channel_metadata`，在通道 metadata 中以 `external_id` 键持久化 `item.device_id`（键值长度/条目数均在 `validate_metadata` 上限内），使 `resolve_gb_command` 能恢复通道协议标识。新增单元测试 `gb_event_sink::catalog::tests::channel_metadata_records_external_id`。

## 附带修复：出站命令传输套接字

Devin Review 指出基线分支出站命令传输缺陷：`run_with_cancellation`（`crates/protocols/cheetah-gb28181-driver-tokio/src/lib.rs`）始终经第一个 UDP socket 发送 `DriverAction::Send`。而 `build_runtime`（`apps/cheetah-signaling/src/assembly.rs`）此前从**首个** GB listener 采用命令总线，与其是否有 UDP bind 无关；若该 listener 仅有 TCP bind，则 `command_socket` 为 `None`，命令被静默丢弃且 warn，而 `OwnerCommandHandler::handle_gb28181_command` 已在 `bus.send` 成功时返回 `Sent`，造成“已派发但从未传输”。

初始修复（本分支）：仅当 listener 具备 `udp_bind` 时才采用其 `command_bus()`，无 UDP listener 时保持总线未装配并拒绝命令而非静默丢弃。合入 `main`（GB4-CMD-003，PR #199）后该缺陷已由上游 `MultiListenerCommandBus`（`apps/cheetah-signaling/src/workers.rs`）彻底修复：`build_runtime` 现按 `listener.id` 收集每个 listener 的 `command_bus()` 至 `gb_command_buses`，并由 `MultiListenerCommandBus` 将命令路由到对应 listener 的 driver。合并时采用上游更完整的按-listener 路由方案，取代本分支较窄的 UDP-only 守卫（若无任何 listener 则不装配总线并告警）。

## 附带修复：`gb_event_sink` 模块拆分
